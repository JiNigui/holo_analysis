# 大文件上传解决方案（1200+文件）

## 问题描述

上传超过1000个文件时出现413错误：
```
werkzeug.exceptions.RequestEntityTooLarge: 413 Request Entity Too Large
```

## 根本原因

Werkzeug（Flask的底层WSGI库）有3个硬编码的限制：

| 限制项 | 默认值 | 作用 |
|--------|--------|------|
| `DEFAULT_MAX_CONTENT_LENGTH` | 无限制 | HTTP请求体总大小 |
| `DEFAULT_MAX_FORM_MEMORY_SIZE` | 16MB | 表单数据在内存中的大小 |
| `DEFAULT_MAX_FORM_PARTS` | **1000** | 表单字段数量（每个文件算1个） |

**关键问题**：这些值是硬编码在 `werkzeug.formparser` 模块中的，**无法**通过 Flask 的 `app.config` 设置！

## 解决方案

### ✅ 方案1：猴子补丁（已实施）

**文件**：`back-end/app/__init__.py:7-11`

```python
# 猴子补丁：在导入时立即修改Werkzeug的全局默认值
import werkzeug.formparser
werkzeug.formparser.DEFAULT_MAX_FORM_MEMORY_SIZE = 3 * 1024 * 1024 * 1024  # 3GB
werkzeug.formparser.DEFAULT_MAX_FORM_PARTS = 2000  # 2000个文件
```

**优点**：
- ✅ 简单有效
- ✅ 全局生效
- ✅ 在应用启动时设置一次即可
- ✅ 支持1200-2000个文件

**缺点**：
- ⚠️ 修改了全局变量（但Flask应用中可接受）

---

### 方案2：分批上传

如果需要上传更多文件（>2000个），可以分批上传：

```javascript
// 前端分批上传
const batchSize = 500  // 每批500个文件
for (let i = 0; i < files.length; i += batchSize) {
  const batch = files.slice(i, i + batchSize)
  await uploadBatch(batch, i === 0)  // 只有第一批清空旧文件
}
```

**优点**：
- ✅ 支持无限数量文件
- ✅ 减少单次请求压力

**缺点**：
- ❌ 需要修改前端代码
- ❌ 上传时间更长
- ❌ 可能出现部分失败情况

---

### 方案3：流式上传

使用分块传输编码（Chunked Transfer Encoding）：

```python
from werkzeug.datastructures import FileStorage

@upload_bp.route('/stream-upload', methods=['POST'])
def stream_upload():
    # 使用流式读取，不一次性加载到内存
    for chunk in request.stream:
        # 处理每个块
        pass
```

**优点**：
- ✅ 内存占用低
- ✅ 支持超大文件

**缺点**：
- ❌ 需要重写上传逻辑
- ❌ 前端需要支持分块上传
- ❌ 实现复杂

---

### 方案4：使用Nginx作为上传代理

在Nginx层面处理文件上传：

```nginx
server {
    client_max_body_size 5G;

    location /upload {
        # 使用nginx-upload-module
        upload_pass /upload-handler;
        upload_store /tmp/uploads;
        upload_store_access user:rw group:rw all:r;
    }
}
```

**优点**：
- ✅ 性能最好
- ✅ 不占用Flask进程

**缺点**：
- ❌ 需要配置Nginx
- ❌ 生产环境才需要

## 为什么app.config['MAX_FORM_PARTS']无效？

Flask的`app.config`只能设置Flask层的配置，**不能**设置Werkzeug层的配置。

Werkzeug在解析请求时，直接读取 `werkzeug.formparser.DEFAULT_MAX_FORM_PARTS` 这个模块级别的全局变量，而不是从Flask配置中读取。

**源码证据**：
```python
# werkzeug/formparser.py
def _parse_multipart(self, stream, mimetype, content_length, options):
    parser = MultiPartParser(
        max_form_memory_size=DEFAULT_MAX_FORM_MEMORY_SIZE,  # 硬编码！
        max_form_parts=DEFAULT_MAX_FORM_PARTS  # 硬编码！
    )
```

因此，必须在导入时修改这些全局变量，这就是"猴子补丁"的由来。
